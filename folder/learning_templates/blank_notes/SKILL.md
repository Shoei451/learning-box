# Note Organizer Skill

ユーザーのノート・資料を分析し、複数の形式の学習コンテンツを生成する。

## 入力の受け取り方

1. **アップロードファイル**: `/mnt/user-data/uploads/` を確認してファイルを読み込む
2. **コンテキスト内テキスト**: ユーザーがペーストしたテキストをそのまま使用
3. **指定なし**: どの資料について作業するか確認する

## 出力モード

ユーザーが明示的に指定した場合はそれに従う。指定がない場合は何を作るか確認する。

### 1. 要約（Summary）

**箇条書き形式:**
```markdown
## 要約：[トピック名]

### 主要ポイント
- ポイント1
- ポイント2

### 重要概念
- **概念A**: 定義・説明
- **概念B**: 定義・説明

### 結論・まとめ
[2〜3文で核心をまとめる]
```

**散文形式:**
段落形式で流れを保ちながら要約。論述系ノートに向いている。

---

### 2. 穴埋めプリント（Fill-in-the-blank）

**キーワード穴埋め**方式：重要語句を `___（　）___` に置き換える。

```markdown
## 穴埋めプリント：[トピック名]

### 問題

1. 光合成は植物の ___（　）___ で行われ、___（　）___ と水から ___（　）___ を合成する。

2. ___（　）___ は細胞の「発電所」と呼ばれ、ATPを生産する。

---

### 答え合わせ

1. 葉緑体 / 二酸化炭素 / 有機物（グルコース）
2. ミトコンドリア
```

**穴埋め選定基準:**
- 固有名詞・専門用語
- 定義の核心部分
- 因果関係のキー要素
- 数値・年号・公式

1ページあたり10〜20問を目安にする。

---

### 3. Q&A形式（Question & Answer）

```markdown
## Q&A：[トピック名]

**Q1. [問い]**
A: [答え]

**Q2. [問い]**
A: [答え]
```

問いの種類を混ぜる：
- 「〜とは何か？」（定義）
- 「なぜ〜なのか？」（理由・因果）
- 「〜と〜の違いは？」（比較）
- 「〜の例を挙げよ」（応用）

---

### 4. マインドマップ構造化（Mind Map）

Markdownの階層リストで表現する。

```markdown
## マインドマップ：[中心トピック]

- **[中心トピック]**
  - サブトピックA
    - 詳細1
    - 詳細2
  - サブトピックB
    - 詳細3
    - 詳細4（→ サブトピックAの詳細1と関連）
```

関連性は `（→ ...と関連）` で示す。

---

### 5. フラッシュカード（Flashcard）

```markdown
## フラッシュカード：[トピック名]

| # | 表（問い） | 裏（答え） |
|---|-----------|-----------|
| 1 | [用語・問い] | [定義・答え] |
| 2 | [用語・問い] | [定義・答え] |
```

---

## ファイル出力

出力はMarkdownファイルとして保存し、`present_files` ツールでユーザーに提示する。

ファイル名の規則：
- `summary_[トピック].md`
- `fill_in_blank_[トピック].md`
- `qa_[トピック].md`
- `mindmap_[トピック].md`
- `flashcards_[トピック].md`

保存先：`/mnt/user-data/outputs/`

---

## 複数モード同時生成

ユーザーが「全部作って」と言った場合、または複数指定した場合は、
すべてのモードを1つのMarkdownファイルにまとめて出力する。

ファイル名：`study_set_[トピック].md`

## インタラクティブHTML出力（オプション）

ユーザーが「HTMLで作って」「インタラクティブにして」「Webページ形式で」と言った場合、
またはより実用的な学習体験を提案したい場合にこのモードを使う。毎回作る必要はない。

### スキルの役割分担

このスキルと `my-html` スキル（または frontend-design スキル）は**協調関係**にある：

| 責任 | note-organizer（このスキル） | my-html / frontend-design |
|------|---------------------------|--------------------------|
| 穴埋めの構造 | ✅ `.blank`、`data-answer`、セクション構成 | — |
| 採点ロジック | ✅ `note.js` のAPI仕様を守る | — |
| ファイル分離 | ✅ HTML/CSS/JS を必ず分離する | — |
| ビジュアルデザイン | — | ✅ 色・フォント・レイアウト |
| アニメーション | — | ✅ トランジション・演出 |

HTMLを生成するとき、**デザインの判断は my-html / frontend-design に委ねてよい**。
ただし以下の構造的制約は必ず守ること。

### 必須：構造的制約（変更不可）

**1. ファイルは必ず3分割する**
```
[トピック名]/
├── index.html   ← コンテンツのみ
├── note.css     ← スタイル（デザインはここに集約）
└── note.js      ← 採点・リセット・ヒントのロジック
```

**2. 穴埋めのHTML構造**（デザインは自由、この属性は必須）
```html
<span class="blank-wrapper">
  <input class="blank" data-answer="正解の語句" placeholder="　　　">
  <!-- ヒントあり の場合のみ追加 -->
  <span class="hint-icon" title="ヒント文">?</span>
</span>
```
- `class="blank"` — note.js が採点対象として認識するために必須
- `data-answer="..."` — 正解文字列（note.js が参照する）
- `class="hint-icon"` — ホバーでヒント表示（省略可）

**3. note.js が依存するID**（削除・変更禁止）
```html
<strong id="score-val">—</strong>   <!-- 正解率の表示 -->
<div class="progress-fill" id="progress"></div>  <!-- プログレスバー -->
```

**4. ボタンのonclick**（削除・変更禁止）
```html
<button onclick="checkAll()">採点</button>
<button onclick="revealAll()">答えを表示</button>
<button onclick="resetAll()">リセット</button>
```

**5. note.js の読み込み**（bodyの末尾に必須）
```html
<script src="note.js"></script>
```

### ワークフロー

```
ノート（MD/PDF） → parse_note.py → blanks.json → HTML生成
```

1. `scripts/parse_note.py` でノートから穴埋め候補を抽出（後述）
2. `blanks.json` の `all_blanks` をセクション別にHTMLに流し込む
3. `note.css` / `note.js` はデザイン変更がなければ既存のものをコピー
4. デザインを変更する場合は `note.css` のみ編集（`note.js` は触らない）

### 出力先とファイル提示

```
/mnt/user-data/outputs/[トピック名]/index.html
/mnt/user-data/outputs/[トピック名]/note.css
/mnt/user-data/outputs/[トピック名]/note.js
```

3ファイルを `present_files` で同時に提示し、
「3ファイルをまとめてダウンロードして同じフォルダに置いてから index.html を開いてください」
と案内する。

## parse_note.py — 自動抽出スクリプト

`scripts/parse_note.py` を使うとノートから穴埋め候補を自動抽出できる。

### ユーザーへのノート記法ガイド（必要に応じて案内する）

| 記法 | 例 | 優先度 | 用途 |
|------|----|--------|------|
| `**太字**` | `**ミトコンドリア**` | 高 | 最重要語句（最もシンプルで推奨） |
| `==ハイライト==` | `==流動モザイクモデル==` | 高 | Obsidian互換の強調 |
| `[語句](blank)` | `[ATP](blank)` | 高 | 明示的な穴埋め指定 |
| `[語句](blank:ヒント)` | `[葉緑体](blank:光合成を行う)` | 高 | ヒント付き穴埋め |
| `` `語句` `` | `` `ATP` `` | 中 | 用語・数式 |
| `~~語句~~` | `~~原核細胞~~` | 低 | 対比・補足 |

### Claudeが呼び出す方法

```bash
python3 scripts/parse_note.py /mnt/user-data/uploads/note.md --out /tmp/blanks.json
```

### 出力JSONの構造（実際の出力形式）

```json
{
  "topic": "細胞と生命の基礎",
  "sections": [
    {
      "heading": "エネルギー産生",
      "sentences": [
        {
          "original": "**ミトコンドリア**は細胞の「発電所」と呼ばれ、`ATP`を合成する。",
          "blanks": [
            {"answer": "ミトコンドリア", "hint": null, "priority": 1, "source": "bold"},
            {"answer": "ATP",            "hint": null, "priority": 2, "source": "code"}
          ]
        }
      ]
    }
  ],
  "all_blanks": [
    {"answer": "ミトコンドリア", "hint": null, "priority": 1, "source": "bold"},
    ...
  ],
  "stats": {"total_blanks": 9, "high_priority": 8, "sections": 3}
}
```

### JSONからHTMLへの変換ルール

`sections[].sentences` を順に処理する：
1. `original` の文字列からマークアップ（`**...**` 等）を除去してプレーンテキストにする
2. `blanks` の各語句の位置を特定し、`<span class="blank-wrapper"><input ...></span>` に置換する
3. `hint` がある場合は `<span class="hint-icon" title="ヒント文">?</span>` を追加する
4. `answer-key` 欄に全語句を `/` 区切りで列挙する

`all_blanks` は Q&A・フラッシュカードセクションの素材として使う。

## 品質基準

- **忠実性**: 元の資料にない情報を追加しない
- **重要度判断**: 全部を均等に扱わず、核心概念を優先する
- **学習しやすさ**: 問いの難易度・粒度が適切か確認する
- **言語**: ユーザーの資料が日本語なら日本語で、英語なら英語で出力（混在も対応）
